# CHANGELOG

## [0.2.0] — Stage 0: Production Readiness Fixes

**Date:** 2026-06-01  
**Scope:** Security hardening, reliability improvements, and UX fixes required before any external user testing.

---

### Fix 1 — Clock Skew Patch

#### What Changed
- **Reduced** the JWT `iat` backdate from 60 seconds to 10 seconds in both:
  - `src/app/api/webhook/route.ts` — the `generateCallToken` monkey-patch used by `connectOpenAi()`
  - `src/modules/meetings/server/procedures.ts` — the user token generated by `generateUserToken()`
- **Added** ±30-second webhook timestamp drift detection with structured logging

#### Why This Approach

**Why we kept the `iat` backdate (reduced to 10s):** Vercel serverless functions run on distributed infrastructure where clock drift between the Vercel host and Stream's token validation servers is a real, persistent issue — not just a local development artifact. Completely removing the backdate causes token rejection ("issued in the future") which silently breaks both user connections and AI agent joining. The original 60-second backdate was too generous; 10 seconds is sufficient to absorb normal NTP drift while minimizing the replay attack window to a negligible interval.

**Why we log drift instead of rejecting on it:** Stream's SDK already validates webhook signatures cryptographically via `streamVideo.verifyWebhook()`. Adding a hard rejection based on timestamp would create false positives during brief NTP resync windows. Instead, we log a warning at >30s drift, giving you observability into any real clock issues without dropping valid webhooks.

**Production readiness:** This change tightens the security window by 6x (60s → 10s) while maintaining reliability. The structured log format (`drift=Xs, event_time=..., server_time=...`) makes it searchable in any log aggregator.

### Fix 2 — Summarisation Feedback Loop

#### What Changed
- **Added** `"failed"` to the `meetingStatus` PostgreSQL enum in `src/db/schema.ts`
- **Added** `FAILED` to the `MeetingStatus` TypeScript enum in `src/constants.ts`
- **Wrapped** the entire Inngest `meetingsProcessing` function in try/catch in `src/inngest/function.ts`
  - On error: transitions meeting to `"failed"` status, then re-throws for Inngest retry tracking
  - On success: transitions to `"completed"` and sends a Stream Chat notification
- **Added** Stream Chat notification on summary completion — the agent sends a message to the meeting's chat channel
- **Added** `refetchInterval` polling (10s) in `src/modules/meetings/ui/views/meeting-id-view.tsx` when status is `"processing"`
- **Created** `src/modules/meetings/ui/component/failed-state.tsx` — error state component
- **Upgraded** `src/modules/meetings/ui/component/processing-state.tsx` — animated loading UI with cycling step indicators
- **Updated** `src/modules/meetings/ui/component/columns.tsx` and `src/modules/meetings/ui/component/status-filter.tsx` — added `failed` status icon, color, and filter option

#### Why This Approach

**Why `failed` status instead of retrying silently:** Silent retries mask real problems. If OpenAI's API is down or a transcript is corrupted, infinite silent retries waste API credits and leave users staring at a "processing" state forever. By transitioning to `"failed"`, the user sees the problem immediately. Inngest still handles retry policies at its own level (via the re-throw), but the user gets a clear signal rather than an infinite spinner.

**Why 10-second polling instead of WebSockets:** React Query's `refetchInterval` is the optimal approach for this use case:
1. **Cost:** WebSocket connections are expensive to maintain at scale. Each open connection holds server memory and a file descriptor. For a status check that happens once per meeting (processing → completed), a polling interval is far more efficient.
2. **Reliability:** WebSocket connections drop on mobile, during tab switches, and behind corporate proxies. Polling with React Query handles all of these gracefully with zero additional code.
3. **Automatic cleanup:** The `refetchInterval` callback checks `query.state.data?.status` — once the status changes to `completed` or `failed`, polling stops automatically. No cleanup logic needed.
4. **Progressive enhancement:** This can be upgraded to WebSockets in Stage 1 (1.1 Real-Time Frontend Updates) without changing the component API. The polling is a reliable bridge.

**Why notify via Stream Chat:** The user already has a chat channel per meeting. Sending the notification through the same channel the user will use to ask follow-up questions keeps everything in one context. No separate notification system needed at this stage.

### Fix 3 — Speaker ID Failure Handling

#### What Changed
- **Added** `lowConfidence` boolean column to the `meetings` table in `src/db/schema.ts`
- **Added** `speakerMappings` table in `src/db/schema.ts` with composite index on `(meetingId, speakerId)`
- **Updated** Inngest processing in `src/inngest/function.ts`:
  - Unmapped speakers get sequential labels (`Participant 1`, `Participant 2`, …)
  - Unmapped speaker IDs are persisted to `speakerMappings` for debugging
  - Confidence ratio is calculated: if >30% of lines are unmapped, `lowConfidence = true`
- **Updated** `getTranscript` tRPC procedure in `src/modules/meetings/server/procedures.ts`:
  - Checks `speakerMappings` for user-provided custom names
  - Falls back to `originalLabel` from the mapping, then to sequential `Participant N`
- **Added** `updateSpeakerName` tRPC mutation in `src/modules/meetings/server/procedures.ts`
  - Validates meeting ownership via `userId` check (authorization)
  - Uses upsert pattern: updates existing mapping or creates new one
  - Input validation via Zod: `name` is required, min 1 char, max 100 chars
- **Added** low-confidence warning banner in `src/modules/meetings/ui/component/completed-state.tsx`
- **Added** inline speaker name editing in `src/modules/meetings/ui/component/transcript.tsx`
  - Edit button only appears on hover for fallback-named speakers
  - Inline input with Enter-to-save, Escape-to-cancel keyboard shortcuts
  - Invalidates transcript query on success for instant UI update

#### Why This Approach

**Why `Participant N` instead of `Unknown`:** Showing "Unknown" for every unmapped speaker makes transcripts unreadable — you can't tell who said what. Sequential labels (`Participant 1`, `Participant 2`) preserve conversational structure. A user reading the transcript can follow the back-and-forth between participants even without knowing their real names.

**Why a separate `speakerMappings` table instead of embedding in the transcript:**
1. **Debugging:** The table serves as a permanent log of which speaker IDs couldn't be resolved. This is invaluable for diagnosing issues — you can query which meetings had unmapped speakers, correlate with Stream's participant events, and identify patterns.
2. **User corrections:** Storing corrections in the same table means the getTranscript procedure only needs one additional query (indexed on meetingId + speakerId) to resolve all overrides. No transcript re-processing needed.
3. **Composite index:** The `(meetingId, speakerId)` index ensures O(log n) lookups regardless of how many meetings accumulate. This is critical for production — without the index, every transcript view would trigger a full table scan.

**Why the 30% threshold for `lowConfidence`:** This threshold balances signal vs. noise:
- Below 30%: A few unmapped speakers is normal (phone dial-ins, browser guests). Flagging these would create alert fatigue.
- Above 30%: More than a third of the conversation is unattributed — the summary quality is meaningfully degraded and the user should know.
- The flag is a boolean rather than a percentage to keep the schema simple and the UI binary (show banner or don't). If granular confidence is needed later, the ratio can be derived from `speakerMappings`.

**Why inline editing instead of a separate page:** Inline editing keeps the user in context. They're already reading the transcript and can see which speakers need correction. A separate "manage speakers" page would require context-switching and make the task feel heavier than it is. The edit pencil only appears on hover for fallback names to avoid visual clutter and prevent users from accidentally editing correctly-identified speakers.

**Authorization model:** The `updateSpeakerName` mutation verifies `meetings.userId === ctx.auth.user.id` before allowing any modification. This prevents one user from editing another user's meeting transcripts even if they somehow obtain the meeting ID.

### Database Migration Required

After deploying these changes, run:
```bash
npx drizzle-kit generate
npx drizzle-kit migrate
```

This will:
1. Add `"failed"` to the `meeting_status` enum
2. Add `low_confidence` boolean column to `meetings` (defaults to `false`, non-nullable — safe for existing rows)
3. Create the `speaker_mappings` table with composite index

All changes are backwards-compatible. Existing meetings with `status = "processing"` will continue to work — they just won't benefit from the new error handling until the next Inngest invocation.
